# .ai/ 폴더 파일 템플릿

이 문서는 프로젝트의 `.ai/` 폴더에 생성할 파일들의 초기 템플릿입니다.
`/apply-standard` 스킬이 표준을 적용할 때 이 템플릿을 참고하여 파일을 생성합니다.

---

## 6. plans/ 폴더

`/deep-plan` 스킬의 최종 산출물이 자동 저장되는 폴더입니다. 수동으로 생성할 필요 없이, `/deep-plan` 실행 시 폴더가 자동 생성됩니다.

**파일명 형식:** `YYYY-MM-DD_HHmm_[작업요약].md`

```markdown
# Deep Plan: [작업 제목]

> 생성일: YYYY-MM-DD HH:mm
> 작업 유형: [보안/UI/인프라/데이터/비즈니스/일반]
> 라운드: N회 (초기 + 수정 N-1회)
> 가중 점수: NN.N / NN.N (NN.N%)
> C6 Hard Gate: 통과 / 실패 / 해당없음
> 상태: 수렴 / 사용자 위임

## 비평 점수

| 관점 | 원점수 | 가중치 | 가중 점수 | 판정 |
|------|--------|--------|----------|------|
| C1. 실현 가능성 | N/10 | ×W | N.N | 통과 |
| C2. 완전성 | N/10 | ×W | N.N | 통과 |
| C3. 과잉 설계 방지 | N/10 | ×W | N.N | 통과 |
| C4. 사용자 의도 부합 | N/10 | ×W | N.N | 통과 |
| C5. 기존 패턴 준수 | N/10 | ×W | N.N | 통과 |
| C6. 보안 | N/10 | ×W | N.N | 통과 |
| C7. 단계 분해 품질 | N/10 | ×W | N.N | 통과 |

### 보안 관문 상태

| 항목 | 값 |
|------|-----|
| 작업 유형 | [태그] |
| C6 Hard Gate 대상 | Y/N |
| C6 원점수 | N/10 |
| 관문 판정 | 통과/실패/해당없음 |

## 요구사항 분석
[명시적/암묵적 요구사항, 범위]

## 영향도 분석
[수정 대상 파일, 의존성, 사이드이펙트]

## 단계별 구현 계획
[전체 단계별 계획]

## 기술적 결정
[선택한 방식, 대안, 트레이드오프]

## 위험 요소
[실패 가능성, 대응 방안]

## 테스트 전략
[검증 방법, 엣지 케이스]

## 비평 이력
### 라운드 1
- 가중 점수: NN.N / NN.N (NN.N%)
- 주요 지적: [요약]

### 라운드 2 (있으면)
- 가중 점수: NN.N / NN.N (NN.N%)
- 개선된 점: [요약]
```

> 이 폴더의 파일은 Git에 커밋하여 팀과 공유합니다. 구현 후 "왜 이렇게 설계했는지" 추적할 때 유용합니다.

## 7. PENDING_PLANS.md

`/deep-plan`이 NightBuilder(무인 자동 개발) 환경에서 수렴하지 못한 보류 계획을 기록합니다. 다음 유인 세션에서 `session-briefing.cjs`가 자동 감지하여 개발자에게 안내합니다.

```markdown
# Deep Plan 보류 건

> 이 문서는 /deep-plan 미수렴 계획을 기록합니다.
> 각 항목을 검토하여 수정/진행/폐기를 결정하세요.

---

## [보류] YYYY-MM-DD HH:mm - [작업요약]
- **상태**: 미수렴 (라운드 N, 가중 비율 NN.N%)
- **C6 Hard Gate**: 통과 / 실패 / 해당없음
- **저장된 계획서**: `.ai/plans/YYYY-MM-DD_HHmm_[작업요약].md`
- **사유**: [정체 / 라운드 초과 / C6 Hard Gate 실패]
- **마지막 비평 요약**: [주요 미달 항목 1~3개]

### 조치 기록
- YYYY-MM-DD: [수정 후 진행 / 현재 계획으로 진행 / 보류 유지 / 폐기]

---
```

> 보류 건이 해결되면 상태를 `[해결됨]` 또는 `[폐기됨]`으로 변경합니다. 해결된 항목은 주기적으로 정리합니다.
